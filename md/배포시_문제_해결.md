## 1. 백엔드 컨테이너 기동 실패 — `WebClient.Builder` 빈 없음

**증상:** AI 연동·프록시 코드에서 `WebClient.Builder` 를 주입받는데, Spring 컨텍스트에 해당 타입의 **Bean 정의가 없어** 애플리케이션이 뜨지 않음(로컬·배포 동일하게 실패 가능).

**원인:** `spring-boot-starter-webflux` 가 classpath에 있어도, **`WebClient.Builder` 는 자동으로 Bean이 등록되지 않는다**고 보는 편이 안전하고, 명시적 `@Bean` 이 없으면 주입 실패.

**해결:** 공용 설정 클래스에 `WebClient.Builder` Bean 을 추가.

```1:14:c:\mini_project_3\medical-service-backend\src\main\java\com\emr\medicare\common\config\HttpClientConfig.java
package com.emr.medicare.common.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.reactive.function.client.WebClient;

@Configuration
public class HttpClientConfig {

    @Bean
    public WebClient.Builder webClientBuilder() {
        return WebClient.builder();
    }
}
```

이후 **이미지 재빌드 → GHCR 푸시 → SSM으로 EC2 재배포** 하면 반영됨.

## 2. SSM 배포 직후 ALB에서 502 (Bad Gateway)

**증상:** Run Command 는 **Success** 인데, 브라우저나 `curl` 로 ALB 주소를 치면 **502**.

**원인:** 타깃 그룹은 EC2:3000 등에 붙어 있는데, **Spring Boot가 완전히 기동하기 전**에는 연결이 거절되거나 헬스가 아직 통과하지 않아 ALB가 502 를 줄 수 있음. 배포 직후 몇 초~수십 초 구간이 특히 그렇다.

**해결:**

- **수십 초 간격으로** `GET /health` (또는 설정한 헬스 경로)를 **여러 번 재시도**.
- ALB 쪽에서는 **헬스 체크 간격·unhealthy threshold** 가 너무 공격적이면 짧은 기동 시간에 불리할 수 있으니, 필요 시 완만하게 조정 검토.
- 여전히 502면 **컨테이너 로그**(`docker logs`)와 **환경 변수**(DB·Redis URL 등)를 먼저 본다 — “워밍업”이 아니라 **앱 크래시**인 경우가 많음.

## 3. SSH(22) 없이 EC2에 배포해야 함

**어려움:** 보안 그룹으로 **SSH 포트가 막혀** 있어 `scp` / `ssh` 로 스크립트·이미지를 넣기 어려운 환경.

**해결:** **AWS Systems Manager Session Manager / Run Command** 만 열려 있으면(HTTPS 443), 로컬에서 `deploy-backend-ssm.ps1`, `deploy-frontend-ssm.ps1`, `deploy-ai-ssm.ps1` 등으로 **원격 셸 스크립트 실행 + env 전달 + docker pull/run** 까지 처리. 인스턴스에는 **`AmazonSSMManagedInstanceCore`** 가 들어간 IAM 인스턴스 프로파일과 **SSM PingStatus=Online** 이 전제.

## 4. GHCR 비공개 이미지 — EC2에서 `docker pull` 실패

**어려움:** GitHub Actions 는 `GITHUB_TOKEN` 으로 GHCR 푸시는 되는데, **EC2의 Docker** 는 별도 로그인이 없으면 private 이미지 pull 이 실패.

**해결:**

- **Secrets Manager**에 예: `medicare/ghcr-read` 처럼 `username` / `token` JSON 저장.
- EC2 역할에 **`secretsmanager:GetSecretValue`** (해당 시크릿 ARN 범위) 허용.
- SSM 배포 스크립트가 시크릿을 읽어 **`docker login ghcr.io`** 후 `docker pull` 하도록 구성(레포 `scripts` 내 주석·구현 참고).

## 5. 백엔드 ↔ AI 연동 — 환경 변수와 ALB DNS

**어려움:** 백엔드가 AI 서비스를 부를 때 **URL이 틀리면** 프록시 엔드포인트만 200이 아니거나 타임아웃.

**해결:** `medical-backend.env` 등에 **AI ALB의 HTTP 베이스 URL**(`AI_BASE_URL` 등 실제 키 이름은 애플리케이션 설정에 맞출 것)을 넣고, SSM으로 `medical-service-infra/medical-backend.env` 등에 반영한 뒤 컨테이너 재기동. 배포 후 **`GET /api/ai/alerts`** 같은 프록시 경로로 **200 + JSON** 이 나오는지 확인해 end-to-end 를 짧게 검증함.

## 6. “배포가 됐는지”를 무엇으로 볼지


| 순서 | 확인 |
| --- | --- |
| 1 | GitHub Actions **초록** + GHCR에 **태그 이미지** 존재 |
| 2 | SSM Run Command **Status=Success** |
| 3 | ALB 타깃 그룹 **healthy** (잠시만 unhealthy 일 수 있음) |
| 4 | 백엔드 **`/health` 200** (필요 시 재시도) |
| 5 | 프론트에서 **로그인·주요 화면** 정상 |
| 6 | (해당 시) **`/api/ai/alerts`** 등 AI 프록시 **200** |
